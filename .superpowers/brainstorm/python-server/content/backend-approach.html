AI 统一调度：三种后端方案

用户的输入同时可能是「记录」或「问题」— AI 怎么判断并路由？

A

统一智能端点 /api/ai/smart

新增一个端点，AI 先判断意图 → 记录则返回解析字段，问答则返回对话文本。前端只调一个接口，逻辑最清晰，后端一次请求搞定。

B

优先解析 → 兜底对话

先调 voice-parse → 如果能识别出记录类型就展示确认卡；如果返回 unknown 则再调 ai-chat。改动最小，不需要新接口，但会有两次请求延迟。

C

并行双调 + 意图权重

同时调 voice-parse 和 ai-chat，AI 在解析结果中附带置信度。置信度高的优先展示，低的丢弃。速度最快但浪费 Token / 费用。

推荐方案 A：统一智能端点

数据流
用户说: "左边喂了15分钟"
↓
POST /api/ai/smart { text: "左边喂了15分钟" }
↓
AI 判断意图 → record
↓
返回: { intent:"record", type:"feeding",
parsed:{ side:"left", duration_minutes:15 }}
↓
前端弹出确认卡片
用户说: "宝宝最近奶量减少了正常吗"
↓
POST /api/ai/smart { text: "宝宝最近奶量..." }
↓
AI 判断意图 → chat
↓
返回: { intent:"chat",
reply:"奶量波动很正常..." }
↓
前端追加到对话流

方案 A 优势

  • 一次请求，无额外延迟
  • 意图判断在后端统一处理
  • 新增端点不影响旧接口，voice-parse 和 ai-chat 保留供 AI 顾问页使用
  • 后端可用 system prompt 精确控制输出格式

注意事项

  • 需要后端新增路由 + AI prompt 工程
  • 需推送到 GitHub + Render 部署
